# AuraPoS Idempotency Guide

## Why Idempotency

Offline POS terminals retry failed requests. Without idempotency, a payment that was actually recorded on the server but whose response was lost in transit would be recorded twice — creating a duplicate order and duplicate charge.

Idempotency keys ensure that submitting the same logical operation N times produces exactly the same result as submitting it once.

---

## Key Format

```
{terminalId}:{timestamp}:{random}

Example: TERM-ABC123-XY9Z8W:1716800000000:aBcD1234
```

| Part | Source | Uniqueness |
|------|--------|------------|
| `terminalId` | `getOrCreateTerminalIdentity()` | Per browser/device |
| `timestamp` | `Date.now()` | Per millisecond |
| `random` | `nanoid(8)` | Collision resistance |

Generated by `generateIdempotencyKey(terminalId)` in `packages/offline/src/idempotency.ts`.

---

## Key Generation Rules

1. **Generate once, reuse on all retries.** Generate the key before the first attempt and store it in the outbox item. Every retry of the same outbox item uses the same `idempotencyKey`.

2. **Never generate a new key for a retry.** A new key would create a new order instead of replaying the existing one.

3. **Never reuse a key for a different operation.** Each logically distinct transaction gets its own key.

4. **Scope is `(tenant_id, idempotency_key)`.** The same key from a different tenant is treated as a different operation.

---

## Frontend Usage

```typescript
// packages/offline/src/idempotency.ts
const key = generateIdempotencyKey(terminal.terminalId);

// Online path — send as header
await fetch("/api/orders/create-and-pay", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "x-idempotency-key": key,
  },
  body: JSON.stringify(payload),
});

// Offline path — store in outbox
await enqueueOutbox({ ..., idempotencyKey: key });
// The sync engine sends the key in the batch payload:
// { idempotency_key: key, ... }
```

## Backend Enforcement

### Online create-and-pay (`POST /api/orders/create-and-pay`)

The backend reads the `x-idempotency-key` header and stores it as `orders.idempotency_key`.

A unique index `(tenant_id, idempotency_key)` on the `orders` table ensures no duplicate can be inserted.

If the key already exists:
- The existing order is returned
- HTTP 200 (not 201) is returned
- Response body is identical to the original response

### Offline batch sync (`POST /api/sync/offline-orders`)

Each order in the batch has its own `idempotency_key`. `SyncOfflineOrder` handles duplicates per-item:
- Duplicate detected → `status: "replayed"` returned for that item
- Client treats `replayed` identically to `synced`

---

## Database Schema

```sql
-- orders table has a unique index:
CREATE UNIQUE INDEX orders_tenant_idempotency_key_unique
  ON orders (tenant_id, idempotency_key)
  WHERE idempotency_key IS NOT NULL;
```

This means even if the application code has a bug, the database prevents duplicate orders at the constraint level.

---

## What Idempotency Does NOT Cover

- **Different operation types.** If the cashier accidentally adds the same item twice to the cart, that creates two distinct orders with two distinct keys.
- **Partial payment retries.** `RecordPayment` uses a row lock (`FOR UPDATE`) on the order to prevent concurrent payment races. Idempotency keys are separate from payment locks.
- **Void/refund operations.** These are separate flows with their own audit trail (planned).

---

## Debugging Duplicate Orders

If a duplicate order is suspected:
1. Search `orders` by `idempotency_key` — if two rows exist, the unique index was not in place at insert time.
2. Check `sync_events` for the `local_order_id` — look for multiple `synced` events.
3. Check `sync_batches` for the terminal's sync history.

The `x-idempotency-key` header is also logged in the API request log (truncated to 80 chars).
