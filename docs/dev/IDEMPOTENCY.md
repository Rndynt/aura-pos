# AuraPoS Idempotency Guide

## Why Idempotency

Offline POS terminals retry failed requests. Without idempotency, a payment that was actually recorded on the server but whose response was lost in transit would be recorded twice — creating a duplicate order and duplicate charge.

Idempotency keys ensure that submitting the same logical operation N times produces exactly the same result as submitting it once.

---

## The Core Problem

```
Terminal                          Network              Server
   │                                │                    │
   │── POST /api/orders ───────────►│──────────────────► │ (creates order, charges customer)
   │                                │  ← response lost   │
   │  ← timeout (5s)                │                    │
   │                                │                    │
   │  (retry) POST /api/orders ────►│──────────────────► │ ???
```

**Without idempotency:** Server creates a second order. Customer is charged twice.

**With idempotency:** Server detects the same key, returns the original response. One order, one charge.

---

## Key Format

```
{terminalId}:{timestamp}:{random}

Example: TERM-ABC123-XY9Z8W:1716800000000:aBcD1234
```

| Part | Source | Uniqueness guarantee |
|------|--------|---------------------|
| `terminalId` | `getOrCreateTerminalIdentity()` — stored in IndexedDB | Per browser/device/tenant |
| `timestamp` | `Date.now()` at key generation time | Per millisecond |
| `random` | `nanoid(8)` | Collision-resistant random suffix |

Generated by `generateIdempotencyKey(terminalId)` in `packages/offline/src/idempotency.ts`.

**Uniqueness scope:** `(tenant_id, idempotency_key)` on the `orders` table. The same key from a different tenant is treated as a separate operation.

---

## Key Generation Rules

1. **Generate once, reuse on all retries.** Generate the key before the first attempt. Store it in the outbox item. Every retry of the same outbox item uses the same `idempotencyKey`.

2. **Never generate a new key for a retry.** A new key would create a second order instead of replaying the existing one.

3. **Never reuse a key for a different operation.** Each logically distinct transaction (one order, one payment) gets its own key.

4. **Generate before first network attempt.** Not after. If the first attempt returns a network error, you need the key to retry — so it must exist already.

---

## Where Keys Are Generated

```typescript
// packages/offline/src/localOrderService.ts
// 1. Generated before any network or IndexedDB write
const idempotencyKey = generateIdempotencyKey(terminalId);

// 2. Stored in IndexedDB outbox item
await enqueueOutbox({
  ...
  idempotencyKey,
  payload: { ..., idempotency_key: idempotencyKey },
});
```

```typescript
// apps/pos-terminal-web/src/hooks/useOfflineOrderSubmit.ts
// Online path — key sent as HTTP header
await fetch("/api/orders/create-and-pay", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "x-idempotency-key": idempotencyKey,
    "x-tenant-id": tenantId,
  },
  body: JSON.stringify(payload),
});

// Offline/fallback path — key stored in outbox, sent during sync
await createLocalOrder({ ..., idempotencyKey });
```

---

## Online Path: Direct API

**Endpoints:**

- `POST /api/orders` for draft/create-only orders.
- `POST /api/orders/create-and-pay` for atomic create-and-pay orders.

**Request header:** `x-idempotency-key: TERM-ABC123-XY9Z8W:1716800000000:aBcD1234`

**Request body fallback:** both endpoints also accept `idempotency_key` in the JSON body. If both are present, the body value is used.

**Backend behaviour:**

```
First request:
  → Reads x-idempotency-key header
  → Inserts order with idempotency_key column set
  → Returns HTTP 201 with new order data

Duplicate request (same key):
  → Reads idempotency key from body or x-idempotency-key header
  → Finds existing order with this tenant-scoped key before inserting
  → Returns HTTP 200 with SAME order data
  → Does NOT create a second order
  → For create-and-pay, does NOT charge again
```

## Offline Path: Batch Sync

Each order in the batch has its own `idempotency_key` field:

```json
{
  "orders": [
    {
      "local_order_id": "abc123",
      "idempotency_key": "TERM-ABC123-XY9Z8W:1716800000000:aBcD1234",
      ...
    }
  ]
}
```

**Backend `SyncOfflineOrder` behaviour:**

```
For each order in batch:
  → Look up existing order by (tenant_id, idempotency_key)
  → If not found: create order → result.status = "synced"
  → If found:     return existing order IDs → result.status = "replayed"

Client treats "replayed" identically to "synced":
  → markOutboxSynced(id)
  → local_orders.update({ serverId, syncStatus: "synced" })
```

## Database Enforcement

```sql
-- orders table — unique constraint prevents duplicates at DB level
CREATE UNIQUE INDEX orders_tenant_idempotency_key_unique
  ON orders (tenant_id, idempotency_key)
  WHERE idempotency_key IS NOT NULL;
```

This means even if application code has a race condition or bug, the database prevents the second insert at the constraint level. The insert will throw a `unique_violation` error, which `SyncOfflineOrder` catches and converts to a `replayed` result.

---

## Payment Idempotency

Payments use a separate idempotency key generated at payment time (not order creation time). The `RecordPayment` use case also uses a **row lock** (`FOR UPDATE`) on the order to prevent concurrent payment races:

```sql
SELECT * FROM orders WHERE id = $1 FOR UPDATE;
-- If already paid: return existing payment
-- If not paid: insert payment + update order status
```

This covers the scenario where two cashiers attempt to pay the same order simultaneously from different terminals.

---

## What Idempotency Does NOT Cover

- **User mistakes.** If a cashier accidentally submits the same order twice by clicking Pay twice in quick succession, the double-submit lock (`inFlightRef` in `useOfflineOrderSubmit`) prevents the second attempt within the same page session. Idempotency prevents it at the server level if both attempts somehow reach the server.

- **Different carts.** If the cashier adds the same item to the cart twice, that is two distinct quantities in one order — not a duplicate order. Idempotency does not apply to cart contents.

- **Refunds and voids.** These are separate flows with their own audit trail (planned Phase 18).

## Debugging Duplicate Orders

If a duplicate order is suspected:

### Step 1 — Check the unique index
```sql
SELECT id, order_number, idempotency_key, created_at
FROM orders
WHERE tenant_id = 'your-tenant'
  AND idempotency_key = 'TERM-ABC123-XY9Z8W:1716800000000:aBcD1234';
```
If two rows exist, the unique index was not in place at insert time. Check migration history.

### Step 2 — Check sync events
```sql
SELECT * FROM sync_events
WHERE tenant_id = 'your-tenant'
  AND local_order_id = 'nanoid-abc123'
ORDER BY created_at;
```
Multiple `synced` events for the same `local_order_id` indicate a replay was not detected.

### Step 3 — Check sync batches
```sql
SELECT * FROM sync_batches
WHERE tenant_id = 'your-tenant'
  AND terminal_id = 'TERM-ABC123-XY9Z8W'
ORDER BY created_at DESC
LIMIT 20;
```
Look for multiple batches from the same terminal within minutes of each other.

### Step 4 — Check frontend outbox
Open browser DevTools → IndexedDB → AuraPoSOfflineDB → sync_outbox. Look for the `local_order_id` — it should have `status: "synced"` on at most one item.

## Performance Considerations

- The unique index on `(tenant_id, idempotency_key)` is partial (WHERE `idempotency_key IS NOT NULL`), so it only covers rows that use idempotency. Legacy orders without keys are unaffected.
- The `idempotency_key` is stored as `text`, not hashed. Maximum key length is 128 characters (validated in the sync endpoint schema).
- On high-volume tenants (>1000 orders/day), the index lookup is O(log N) — negligible overhead.

---

## Key Collision Risk

**Theoretical probability of collision:**

- `nanoid(8)` with alphabet of 64 characters → 64^8 ≈ 2.8 × 10^14 unique values.
- Even with 1,000 orders/day from a single terminal, reaching a collision would require ~790 million years.

**Real risk:** Clock skew (same millisecond) + `nanoid` collision. Probability: ~1 in 10^12 per order. Not mitigated by application logic — the unique index constraint catches it if it ever occurs.

---

## Future: Idempotency for Other Operations

Currently, idempotency keys are only enforced on order creation. Future sprints will extend to:

- `POST /api/orders/:id/refund` — `x-idempotency-key` header
- `POST /api/orders/:id/void` — `x-idempotency-key` header
- `POST /api/payments` (standalone payment) — `x-idempotency-key` header

Each endpoint will store the key with a `(tenant_id, operation_type, idempotency_key)` index.
